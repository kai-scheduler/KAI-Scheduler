New preemption mode · Design proposal

Semi-Preemptible PodGroups

A third preemption mode that lets a workload guarantee a minimum core of protected pods while bursting opportunistically with elastic pods that can be reclaimed at any time. One PodGroup — assured capacity and on-demand scale.

The gap

Today it's all-or-nothing

Since v0.10, Priority and Preemption are independent — but Preemption has only two modes. Real AI workloads want both behaviors at once.

🔒

Non-preemptible

Every pod is protected. Safe, but greedy — it can never give back idle capacity, even temporarily.

💨

Preemptible

Every pod can be reclaimed. Flexible, but offers zero guarantee — unsuitable for critical services.

🧩

Semi-preemptible new

Non-preemptible up to minMember per leaf PodSet; every extra pod is elastic and reclaimable.

Use case. Inference and elastic distributed training with a minReplicas: run the critical minimum with assured resources, and let everything above it ride on availability-based, on-demand capacity.
The model

Core pods are locked. Extra pods float.

The split is decided by a single, familiar threshold: minMember on each leaf PodSet. Pods at or below it are core; pods above it are elastic.

A leaf PodSet with minMember = 3, running 5 pods — sorted by preemption priority →
🔒corerank 1
🔒corerank 2
🔒corerank 3
💨 elasticrank 4
💨 elasticrank 5
Core — the top minMember by order: in-quota, protected from eviction Elastic — everything below the line: may be over-quota, reclaimed first

The "core/elastic" line is positional, not an identity: it's a threshold over an ordering, not a tag on a specific pod. The guarantee is "at least minMember survive," never "this exact pod is safe."

total pods = minMember  →  behaves like a non-preemptible job
minMember = 0  →  behaves like a preemptible job
The key mechanism

How do we identify the elastic pods?

We don't tag them. There is no per-pod "elastic" flag. The split is derived dynamically every scheduling cycle by two independent, count/order-based mechanisms — both reusing logic that already exists.

🗑️

Eviction — which pods are reclaimable

At a leaf PodSet: if active pods ≤ minMember, protect everything. Otherwise sort all active pods by the scheduler's standard task-order comparator and evict from the bottom.

Elastic set = the lowest-ranked count − minMember pods, computed live. Reuses the existing minAvailable eviction guard — no new logic.

🧮

Quota — which pods count against quota

A running counter marks the first minMember allocated pods per PodSet as core (AllocatedNotPreemptible).

Identity is irrelevant here — only the aggregate resource quantity feeds the quota check, so any minMember pods give the same number.

Consequences worth naming

!

Determinism rides on the task-order comparator

If two pods tie under the ordering (same priority, no stable tiebreaker), which one floats can flip between cycles → core-membership churn. A stable final tiebreaker (e.g. UID) keeps the set steady.

≡

Quota counting assumes homogeneous pods per PodSet

"First minMember pods" yields a well-defined core resource sum only when pods in the set request the same resources. Heterogeneous PodSets make the core quota ambiguous.

👁️

No per-pod observability

Nothing on the pod tells an operator its tier — it's an emergent scheduler property. Adding a label/condition is exactly the work the Phase 2 third tier would require.

Quota

Core counts against quota. Elastic doesn't.

Core pods must be in-quota when allocated. Extra pods may be allocated over-quota — but all pods still respect the queue's hard Limit.

✅

Core → quota

Only core pods accrue to the queue's AllocatedNotPreemptible. A core pod is allocated only if it fits within deserved quota.

📈

Elastic → over-quota OK

Extra pods can borrow unused cluster capacity beyond the queue's quota — opportunistically.

⛔

Limit → hard ceiling

Total allocation (core + elastic) never exceeds the queue's MaxAllowed limit.

Edge case: quota scale-down

If a queue's quota drops below a running job's core allocation, reclaim evicts the elastic pods first; core pods stay (protected by the existing minAvailable guard) and the queue sits over-quota until the job ends. This is accepted — identical to how a fully non-preemptible job behaves today. Conditional-reclaim and a queue quota-floor webhook are recorded as deferred future options.
Subgroups & multi-level trees

Elasticity is a pod-level concept

↳

Subgroups are atomic

There is no "semi-elastic subgroup". A subgroup is scheduled as a whole — the core/elastic split applies only to pods.

↳

Mode is inherited

If the root PodGroup is semi-preemptible, all of its subgroups are semi-preemptible.

↳

minMember vs minSubGroups

The split is decided only at leaf PodSets that set minMember. Intermediate minSubGroups nodes are a scheduling gate — they don't define a non-preemptible pod threshold.

Σ

Non-preemptible resource count

sum( minMember × pod request ) across all scheduled leaf PodSets.

Constraints

Immutability keeps quota honest

A validating webhook enforces these on semi-preemptible PodGroups.

!

No increasing minMember / minSubGroups after creation

Growing the threshold would silently reclassify already-running over-quota elastic pods as protected core pods — violating quota invariants without a rescheduling cycle. Applies to the root spec and every subgroup.

✓

Decreasing is allowed

Lowering the threshold can only widen the elastic tier — always safe.

≈

Simulation considers only the extra pods as victims

Per leaf PodSet, only n − minMember pods are candidate victims, applied independently per PodSet. Simple and sufficient for the MVP.

Implementation

Small, surgical, opt-in

Zero impact when no semi-preemptible jobs exist. Core-pod protection reuses logic that already exists.

AreaChange
API semi-preemptible added to the Preemptibility enum; kubebuilder validation marker updated.
Victim selection Job-level filter now admits semi-preemptible jobs into victim pools. Core protection handled downstream by the existing minAvailable eviction guard — no new eviction logic.
Quota check The non-preemptible quota check uses a coreRequiredQuota (≤ minMember per PodSet); the MaxAllowed check still uses the full request.
Quota accounting Allocate/dealloc handlers track AllocatedNotPreemptible per task — a task is core if its PodSet's allocated count is ≤ minAvailable. Session init seeds counts from already-running jobs.
Reclaim performance. A semi-preemptible job contributes at most n − minMember elastic victims — bounded by a fully-preemptible job of that smaller size. No combinatorial blow-up; the feature is inherently opt-in.
Future work

Decoupling with minNonPreemptible phase 2

Today minMember is both the scheduling minimum and the non-preemptible threshold. A future minNonPreemptible field would separate them — e.g. minMember = 4, minNonPreemptible = 2: needs 4 pods to start, but only 2 are protected.

Key complexity: introduces a third pod tier — "required to schedule, but elastic for preemption" — between core and extra-elastic. Pod ordering today has two tiers; this adds a third, requiring explicit tier identification. Deferred out of the MVP.
